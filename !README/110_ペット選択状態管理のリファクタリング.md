# 110. ペット選択状態管理のリファクタリング

## 概要

このリファクタリングでは、アプリケーション全体で「選択中のペット」の状態管理方法を統一し、コンポーネント間のデータフローを改善しました。

## 変更前の問題点

以前の実装では、選択中のペットの状態 (`selectedPetId`) が `usePets` フックの内部で管理されていました。また、`Header` コンポーネントがその状態を変更する唯一の場所でした。

これにより、以下の問題が発生していました。

-   **状態の不整合**: `Home` ページ (`page.tsx`) や他のコンポーネントが、どのペットが選択されているかを直接知ることができませんでした。
-   **複雑なデータフロー**: `useLogbook` フックが `usePets` フックに暗黙的に依存しており、データの流れが分かりにくくなっていました。
-   **再利用性の低下**: `Header` コンポーネントが `usePets` フックと密結合していたため、他のページ (`history/page.tsx`, `settings/tasks/page.tsx`, `pets/page.tsx` など) で再利用する際に、同様のロジックを再度実装する必要があり、実際に複数ページで実行時エラーが発生しました。

## 変更内容

この問題を解決するため、Reactのベストプラクティスである「Stateのトップダウン管理（Lift State Up）」に従い、以下のようにリファクタリングを行いました。

### 1. ページコンポーネントでの状態管理集約

-   選択中のペットの状態 (`selectedPet`) を `useState` を使い、`Header`コンポーネントを利用する各ページ (`page.tsx`, `history/page.tsx`, `settings/tasks/page.tsx`, `pets/page.tsx`) で一元管理するように変更しました。
-   `usePets` フックからペットのリスト (`pets`) を取得し、`selectedPet` の初期値を設定します。

### 2. propsによるトップダウンのデータフロー

-   親コンポーネントから、子コンポーネントへ必要な情報 (`selectedPet`) と状態更新関数 (`setSelectedPet`) を `props` として渡すようにしました。
    -   **`Header`**: `pets`, `selectedPet`, `onPetChange` を受け取るように変更。これにより、`Header` は状態を持たない純粋なUIコンポーネントになりました。
    -   **`TaskSelector`**: `selectedPet` を受け取るように変更。
    -   **`LogTimeline`**: `selectedPet` を受け取るように変更。
    -   **`ManualAddLogModal`** / **`EditLogModal`** / **`TaskFormModal`**: `selectedPet` を受け取るように変更。

### 3. カスタムフックの修正

-   **`usePets`**: `selectedPetId` の状態管理を削除し、ペットリストの取得のみに責務を限定しました。
-   **`useLogbook`**: `usePets` への依存をなくし、引数として `petId` を受け取るように変更しました。これにより、特定のペットのログやタスクを明示的に取得できるようになりました。

### 4. 関連ページへの修正適用

-   `page.tsx` と同様のロジックを、`Header`コンポーネントを利用するすべての関連ページ (`history/page.tsx`, `settings/tasks/page.tsx`, `pets/page.tsx`) に適用しました。
-   各ページで `selectedPet` の状態を管理し、`Header` やその他のコンポーネントに適切な `props` を渡すように修正しました。
-   これにより、コンポーネントの再利用時に発生していた実行時エラーが解消され、アプリケーション全体で状態管理が統一されました。

## 変更によるメリット

-   **単一方向のデータフロー**: データが親から子へ一方向に流れるようになり、アプリケーションの状態が予測しやすくなりました。
-   **コンポーネントの再利用性向上**: `Header` などのコンポーネントが特定のフックに依存しなくなり、再利用性が向上しました。
-   **保守性の向上**: 状態管理がページ単位で明確に分離されたことで、将来の機能追加やデバッグが容易になりました。
