# 112_家族と共有機能実装手順

このドキュメントは、犬ライフログアプリ「aina-Life」における「家族と共有機能」の実装要件とロードマップを整理したものです。

---

## 1. 機能概要

ユーザーが所有するペットの情報を、他のユーザーと共有し、共有されたペットに対してロールベースのアクセス制御を行う機能です。

---

## 2. 要件整理

ユーザーから提示された要件を整理し、実装の複雑性を考慮して分類します。

### 2.1. コア機能要件 (必須)

-   **ペット情報の共有**:
    -   あるユーザーが所有するペットの情報を、他の特定のユーザーと共有できるようにする。
    -   共有されたペットは、共有先のユーザーのペット一覧にも表示される。
-   **ロールベースのアクセス制御**:
    -   共有されたペットに対して、ユーザーごとに異なる権限を設定できる。
    -   **`owner` ロール**:
        -   ペットの作成者、または共有によって「所有者」権限を与えられたユーザー。
        -   ペット情報の閲覧、編集、削除が可能。
        -   ペットに紐づくログの閲覧、追加、編集、削除が可能。
        -   ペットに紐づくタスクの閲覧、追加、編集、削除が可能。
        -   他のユーザーをペットの共有に招待、共有ユーザーのロール変更、共有解除が可能。
    -   **`general` ロール**:
        -   ペット情報の閲覧、編集が可能。
        -   ペットに紐づくログの閲覧、追加、編集、削除が可能。
        -   ペットに紐づくタスクの閲覧、追加、編集が可能。
        -   他のユーザーをペットの共有に招待することが可能（ユーザー招待時はデフォルトで閲覧者）。
        -   `viewer`ロールのユーザーを`general`ロールに変更することが可能。
        -   **制限**: ペット自体の削除、タスクの削除、他のユーザーの共有解除（自身以外）は不可。
    -   **`viewer` ロール**:
        -   ペット情報の閲覧が可能。
        -   ペットに紐づくログの閲覧が可能。
        -   ペットに紐づくタスクの閲覧が可能。
        -   他のユーザーをペットの共有に招待することが可能。
        -   **制限**: ペット情報の編集・削除、ログの追加・編集・削除、タスクの追加・編集・削除、共有ユーザーのロール変更、他のユーザーの共有解除は不可.

### 2.2. ユーザー管理要件 (必須)

-   **共有ユーザーの招待**:
    -   ペットのオーナーが、他のユーザーをメールアドレス（またはユーザーID）でペットの共有に招待できるUI。
    -   招待されたユーザーが、そのペットにアクセスできるようになる。
-   **共有ユーザーの管理**:
    -   ペットのオーナーが、共有しているユーザーの一覧を確認できるUI。
    -   共有しているユーザーのロール（`owner` / `general`）を変更できるUI。
    -   共有しているユーザーとの共有を解除できるUI。

### 2.3. UI/UX 要件 (必須)

-   **設定画面への統合**:
    -   ヘッダーの既存の設定ボタンからアクセスできる、専用の「設定」画面（`/settings`）を設ける。
    -   この設定画面内に、ユーザー自身の情報管理セクションと、ペット共有管理セクションを配置する。
-   **ユーザー情報の表示**:
    -   設定画面で、ログイン中のユーザーの表示名とメールアドレスを確認できる。
-   **ログアウト機能**:
    -   設定画面からログアウトできるボタンを配置する。
-   **各画面での権限反映**:
    -   ホーム、履歴、タスク管理、ペット管理の各画面で、現在のユーザーが選択中のペットに対して持つロールに応じて、UI要素（ボタン、入力フィールドなど）の表示/非表示、有効/無効を適切に制御する。

### 2.4. 高度な要件 (オプション/将来検討)

-   **`admin` ロール (グローバル管理者)**:
    -   システム内の**全てのペット**を管理できる特別なロール。
    -   **複雑性**: システム全体のユーザー管理、セキュリティルールの大幅な複雑化、専用の管理UIが必要となるため、今回の「家族と共有機能」の範囲を超えた、独立した大きな機能となります。
-   **招待の承認フロー**:
    -   招待されたユーザーが、共有を承諾/拒否するプロセス（例: 招待メール、アプリ内通知）。
-   **共有中のペットの表示**:
    -   ユーザーが共有されているペットを、自分のペット一覧とは別に識別できるUI（例: アイコン表示、ラベル）。
-   **共有解除時のデータ処理**:
    -   共有が解除された際に、共有先のユーザーがそのペットのデータにアクセスできなくなることを保証する。

## 3. 必要最低限の実装範囲 (提案)

上記の要件整理を踏まえ、**「家族と共有機能」の必要最低限の範囲**として、以下を提案します。

### 3.1. 提案する範囲

-   [ ] **ペット情報の共有 (コア機能)**: ユーザーが所有するペットを、他のユーザーと共有できるようにする。共有されたペットは、共有先のユーザーのペット一覧に表示される。
-   [ ] **ペット固有のロールベースのアクセス制御**:
    -   **`owner` ロール**: ペットのCRUD、ログのCRUD、タスクのCRUD、共有管理が可能。
    -   **`general` ロール**: ペット情報の閲覧・編集、ログのCRUD、タスクの閲覧・追加・編集が可能。`viewer`ロールのユーザーを`general`ロールに変更することが可能。**制限**: ペット自体の削除、タスクの削除、他のユーザーの共有解除（自身以外）は不可。
    -   **`viewer` ロール**: ペット情報の閲覧、ログの閲覧、タスクの閲覧が可能。他のユーザーをペットの共有に招待することが可能。**制限**: ペット情報の編集・削除、ログの追加・編集・削除、タスクの追加・編集・削除、共有ユーザーのロール変更、他のユーザーの共有解除は不可。
-   [ ] **共有ユーザーの招待と管理 (UI)**: ペットのオーナーが、他のユーザーをメールアドレスで招待できるUI。共有しているユーザーの一覧表示と、共有解除できるUI。
-   [ ] **UI/UXの適応**: 設定画面への統合、ユーザー情報表示、ログアウト機能。各画面で、`owner` と `general` のロールに応じたUI制御（ボタンの有効/無効、表示/非表示）。

### 3.2. 今回のフェーズから除外されるもの

-   **`admin` ロール (グローバル管理者)**: これは非常に複雑なため、今回のフェーズでは含めません。
-   **招待の承認フロー**: シンプル化のため、招待したら即座にアクセス可能とする（または、招待されたユーザーがアプリにログインした際に自動的に共有ペットが表示される）。
-   **共有中のペットの特別な表示**: UI上の識別は後回し。
-   **共有解除時のデータ処理の複雑化**: Firestoreセキュリティルールでアクセスを制限するのみ。

### フェーズ2: フロントエンド - コアロジックと共有管理UI

1.  **`usePets.ts` の共有ロジック強化:**
    *   **関数:** `inviteUserToPet(petId: string, invitedUserEmail: string, role: "owner" | "general")`: ユーザーを招待し、`sharedUsers` サブコレクションにドキュメントを追加。

### フェーズ3: フロントエンド - UI の適応と改善

1.  **ペット管理UI (`src/app/pets/page.tsx`) の適応:**
    *   **目標:** `general` ロールのユーザーに対して「削除」ボタンを無効化/非表示にする。
    *   **アクション:** ユーザーのロールに基づいて削除ボタンを条件付きでレンダリング/無効化するように `src/app/pets/page.tsx` を変更します。
2.  **タスク管理UI (`src/app/tasks/page.tsx`) の適応:**
    *   **目標:** `general` ロールのユーザーのタスクCRUD操作を無効化/非表示にする（またはログの追加のみを許可し、タスクの管理は許可しない）。
    *   **アクション:** ユーザーのロールに基づいてタスク管理ボタンを条件付きでレンダリング/無効化するように `src/app/tasks/page.tsx` を変更します。
3.  **ログ管理UI (`src/app/history/page.tsx`) の適応:**
    *   **目標:** `general` ロールのユーザーがログを追加できるようにするが、他のユーザーが作成したログの編集/削除は制限する。
    *   **アクション:** ユーザーのロールとログの `createdBy` フィールドに基づいて編集/削除ボタンを条件付きでレンダリング/無効化するように `src/app/history/page.tsx` と `src/components/LogTimeline.tsx` を変更します。
4.  **ユーザープロファイル管理 (オプションの強化):**
    *   **目標:** ユーザーが `displayName` やその他のプロファイル情報を編集できるようにする。
    *   **アクション:** `src/app/settings/page.tsx` と `useAuth.ts` にUIとロジックを実装します。
