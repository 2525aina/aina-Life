# aina-Life 開発総合ガイド

このドキュメントは、犬ライフログアプリ「aina-Life」の開発に関するすべての情報を集約した総合的なガイドです。新しい開発者は、まずこのドキュメントを読むことで、プロジェクトの全体像から具体的な実装ルールまでを把握することができます。

---

## 1. プロジェクト概要（要件）

### 1.1. アプリの目的
犬の1日の行動を簡単に記録・管理し、複数ユーザー（家族など）での共有を可能にするライフログアプリケーションです。

### 1.2. 主要機能
- 分単位での時間入力と、タスクボタンによるリアルタイム記録
- 日付の切り替え、ログの編集・削除
- 複数ペットのプロフィール管理と、複数ユーザーでのログ共有
- Googleアカウントによる簡単認証
- オフラインでのログ記録とオンライン復帰時の自動同期
- （将来拡張）ログの分析・統計機能

---

## 2. アーキテクチャと技術スタック

### 2.1. システム構成
Firebaseの各種サービスを最大限に活用し、サーバーレスでスケーラブルな構成を目指します。

- **フロントエンド**: Next.js + Firebase Hosting
- **データベース**: Firestore
- **バックエンドロジック**: Firebase Functions (Node.js)
- **認証**: Firebase Authentication
- **クラッシュレポート**: Crashlytics

### 2.2. 基本技術スタック
- **フレームワーク**: Next.js (App Router)
- **言語**: TypeScript
- **UIライブラリ**: React
- **バックエンドサービス**: Firebase
- **スタイリング**: Tailwind CSS

```json

  "name": "string",
  "email": "string",
  "createdAt": "Timestamp"
}
```

### 3.2. `dogs` コレクション
```json
dogs/{dogId} {
  "ownerIds": ["userId1", "userId2"],
  "name": "string",
  "breed": "string",
```

`dogs/{dogId}`のサブコレクションとして定義します。
```json
dogs/{dogId}/tasks/{taskId} {
  "name": "string",
  "color": "string",

  "createdAt": "Timestamp"
}
```

```json

  "inputType": "auto" | "manual",
  "note": "string",
  "createdBy": "userId",
  "createdAt": "Timestamp",
```

## 4. 画面設計

### 4.1. 主要画面
| 画面 | URLパス | 目的 |
|---|---|---|
| **ログイン** | `/login` | Googleアカウントによる認証。 |
| **ホーム** | `/` | 行動のワンタップ記録と当日のログ確認。 |
| **履歴** | `/history` | 過去ログの確認・管理。 |
| **タスク管理** | `/settings/tasks` | ログ用タスクのカスタマイズ。 |
| **ペット管理** | `/pets` | ペット情報の管理と共有設定。 |

### 4.2. ホーム画面 (`/`) コンポーネント設計
- **`<Header />`**: ペット選択、設定への導線。
- **`<TaskSelector />`**: タスクボタンを横一列に表示し、ワンタップでログを記録。
- **`<LogTimeline />`**: 今日のログを時系列で表示。
- **`<QuickAddButton />`**: 手動入力モーダルを開くためのボタン。
- **`<ManualAddLogModal />`**: 過去時刻のログやメモ付きログを記録。
- **`<FooterNav />`**: 主要画面へのナビゲーション。

*(詳細は `03_設計.md` を参照)*

## 5. コーディング規約と設計思想

開発における品質と一貫性を保つためのルールです。

### 5.1. ロジックの分離：カスタムフックの徹底活用
- **指針**: 「ユーザー認証」や「データ取得」のようなビジネスロジックは、UIコンポーネントから完全に分離し、`src/hooks`ディレクトリに**カスタムフック**としてカプセル化します。
- **依頼例**: 「機能ロジックは、コンポーネント内に直接書かず、`src/hooks`ディレクトリにカスタムフックとして切り出してください。」

### 5.2. コンポーネントの責務：自己完結型コンポーネント
- **指針**: 各コンポーネントは、親から指示（Props）を待つだけでなく、**自身が必要とするデータや関数を、自身でフックを呼び出して取得**します。これにより、コンポーネントの再利用性が高まり、疎結合な設計になります。
- **依頼例**: 「コンポーнентは、Propsでデータを受け取るだけでなく、必要に応じて自身でカスタムフックを呼び出し、自己完結するように実装してください。」

### 5.3. 状態管理：React標準機能の活用
- **指針**: グローバルな状態管理のために、すぐにReduxのような外部ライブラリを導入せず、まずはReact標準の`useState`, `useEffect`, `Context`（フック経由で利用）で実現することを基本方針とします。
- **依頼例**: 「状態管理は、まずReactの標準機能でシンプルに実装することを検討してください。」

### 5.4. ファイル構成規約
- **指針**: 以下の規約に従うことで、プロジェクトの見通しを良くします。
  - `src/app`: Next.jsのルーティング（ページ）定義
  - `src/components`: 再利用可能なUI部品
  - `src/hooks`: 機能ロジックをまとめたカスタムフック
  - `src/lib`: Firebaseの初期化など、ライブラリ関連の設定
- **依頼例**: 「既存のディレクトリ構成に従って、適切な場所にファイルを作成してください。」
